# 报销单填写自动化 PRD

## 1. 文档目的

本文档用于定义 `all-in-one_automation` 仓库中的下一条业务 flow：`报销单填写`。

该 flow 的定位不是仅支持单一报销类型，而是：

- 在当前浏览器自动化基座之上，新增一条可扩展的“报销单填写”业务 flow
- 首个落地场景以“费用报销类 - 业务招待费报销”为样板场景
- 后续在不推翻现有结构的前提下，逐步扩展到其他报销类型

本文档聚焦：

- 业务目标与范围
- 页面流程与关键节点
- 输入文件与配置口径
- 自动化基座与业务 flow 的边界
- v0.1 首版实现范围

---

## 2. 背景与目标

当前仓库已经完成：

- 通用浏览器自动化基座
- 档案材料上传 flow
- `automation/ + flows/ + runtime/` 的稳定结构

下一步需要基于现有基座承接新的业务场景，以验证：

- 基座是否能够支持更复杂的导航型业务流程
- 新业务 flow 的接入是否可以遵循统一的任务文件、配置文件、运行目录与 CLI 规范

`报销单填写` 被选为下一条 flow，原因是：

- 流程完整，包含登录后导航、菜单进入、单据创建、附件上传、字段填写、明细填写、暂存保存等典型自动化步骤
- 与当前材料上传 flow 有一定复用，也引入了新能力，如菜单导航、tab 切换、明细区填写、多附件识别等
- 适合作为自动化基座扩展能力的下一块真实业务样板

本 PRD 的首要目标不是一次性覆盖所有报销单类型，而是：

`在通用报销单填写框架下，先跑通“业务招待费报销”场景。`

---

## 3. Flow 定位

### 3.1 Flow 名称

建议 flow 名称：`reimbursement-fill`

CLI 示例：

```powershell
python .\main.py validate reimbursement-fill --config .\config\reimbursement_fill.yaml --tasks .\runtime\reimbursement_fill\task.xlsx
python .\main.py run reimbursement-fill --config .\config\reimbursement_fill.yaml --tasks .\runtime\reimbursement_fill\task.xlsx --headed
```

### 3.2 Flow 边界

该 flow 负责：

- 进入 IAM 面板
- 跳转至财务共享系统
- 导航到网上报账平台
- 创建指定报销类型单据
- 上传并识别发票附件
- 填写单据信息与报销明细
- 点击保存，完成暂存

该 flow 当前不负责：

- 提交流程审批
- 审批流节点流转
- 报销单修改、撤回、冲销
- 多报销类型共用逻辑之外的复杂特例处理
- OCR 结果校验与发票内容比对

## 4. v0.1 首版场景定义

### 4.1 首个落地场景

`费用报销类 - 业务招待费报销`

### 4.2 首版业务目标

在真实系统中自动完成以下动作：

1. 默认用户已处于 IAM 登录态，进入 IAM 面板
2. 点击“财务共享”
3. 进入“财务共享 - 网上报账平台”
4. 点击“我要报账”
5. 新建“费用报销类 - 业务招待费报销”单据
6. 进入“电子影像”页签/页面
7. 上传本地 PDF 发票文件
8. 对每张发票执行“识别”
9. 回到报销单主页面
10. 填写单据信息
11. 填写报销明细信息
12. 点击“保存”，完成暂存

### 4.3 首版成功标准

满足以下条件视为单条任务成功：

- 成功进入“业务招待费报销”填报页面
- 该任务关联的所有发票文件均上传成功
- 每张发票均完成一次“识别”动作
- 单据信息字段填写完成
- 明细信息填写完成
- 点击“保存”后页面出现保存成功提示，或页面状态表明已暂存成功
- 结果写回 `runtime/reimbursement_fill/task.xlsx`

## 5. 页面流程梳理

基于当前《报销单填报步骤.pdf》，首版流程梳理如下。

### 5.1 入口流程

1. 默认用户已处于 IAM 登录态并打开 IAM 面板
2. 点击 IAM 看板中的“财务共享”
3. 进入财务共享系统

### 5.2 系统导航流程

1. 点击左侧菜单树中的“财务共享 - 网上报账平台”
2. 点击右侧“我要报账”按钮
3. 点击左上角“新建单据”
4. 选择“费用报销类 - 业务招待费报销”

### 5.3 电子影像上传流程

1. 点击“电子影像”
2. 在电子影像上传维护页点击“本地上传”
3. 在弹框中点击“选择文件”
4. 选择本任务关联的本地 PDF 发票文件并打开
5. 点击“开始上传”
6. 上传完成后关闭上传弹框

### 5.4 发票识别流程

对该任务下的每张发票循环执行：

1. 在左侧列表中选中对应发票文件
2. 点击右侧“识别”按钮
3. 等待识别完成

### 5.5 单据信息填写流程

回到“业务招待费报销”主页面，填写：

- 附件个数：由 `invoice` sheet 中同一 `task_id` 的发票行数自动计算
- 业务单位：来自 `task` sheet
- 付款用途：来自 `task` sheet

### 5.6 报销明细填写流程

点击“报销明细信息”tab，对每条发票明细填写：

- 接待类型：固定“内部招待”
- 公司人数：来自 `invoice` sheet
- 备注：来自 `invoice` sheet

当前根据你的步骤文档，首版默认一条 `invoice` 记录对应一条报销明细。

### 5.7 保存流程

1. 点击右上角“保存”
2. 等待保存成功提示
3. 记录结果并结束当前任务

## 6. 输入文件设计

### 6.1 运行目录建议

建议延续当前仓库结构：

```text
runtime/
└─ reimbursement_fill/
   ├─ task.xlsx
   └─ file/
```

### 6.2 task.xlsx 首版结构

首版采用双 sheet 结构：

- `task`：主单信息
- `invoice`：发票明细信息

#### sheet: task

建议字段如下：

- `task_id`
  - 任务唯一编号
- `business_department`
  - 业务单位
- `payment_purpose`
  - 付款用途
- `upload_result`
  - 结果回写列

#### sheet: invoice

建议字段如下：

- `task_id`
  - 关联到 `task.task_id`
- `company_count`
  - 公司人数
- `file_path`
  - 发票文件路径
- `remark`
  - 备注

### 6.3 首版设计约束

- v0.1 即支持单任务下多发票文件
- `attachment_count` 不单独存储，而是由 `invoice` sheet 中同一 `task_id` 的记录数动态计算
- `task` 与 `invoice` 必须通过 `task_id` 进行关联
- 每个 `task_id` 至少要有一条 `invoice` 记录

## 8. 自动化基座与 Flow 的边界

### 8.1 应继续下沉到基座的能力

如果后续在报销 flow 中确认稳定且可复用，建议继续沉到 `automation/` 的能力包括：

- 左侧菜单树导航
- tab 切换
- 多文件上传
- 列表逐项处理
- 成功提示等待
- 复杂表单明细追加/编辑

### 8.2 先保留在 reimbursement_fill flow 内的能力

首版建议先保留在该 flow 内的逻辑：

- “费用报销类 - 业务招待费报销”的单据创建路径
- 单据信息字段填报顺序
- 明细信息填写规则
- 发票识别流程顺序

原因是这些逻辑当前仍强依赖具体业务页面，不宜过早抽象。

一句话原则：

`先让报销单填写主链路跑通，再把其中重复且稳定的能力继续下沉到基座。`

## 9. 异常处理口径

首版沿用当前项目既有口径：

- 每一步失败后截图
- 记录失败原因
- 写日志
- 写 SQLite
- 当前任务标记失败
- 继续下一条任务
- 暂不实现人工确认后继续
- 暂不实现自动重试

### 9.1 首版常见失败点

预计包括：

- IAM 面板入口加载失败
- 财务共享入口点击失败
- 左侧菜单点击失败
- “新建单据”类型选择失败
- 上传弹框打开失败
- 文件上传失败
- 发票识别失败
- 单据字段填写失败
- 明细填写失败
- 保存失败
- 保存成功提示未出现

## 10. v0.1 开发顺序建议

建议按以下顺序实现：

1. 打通 IAM 面板与财务共享入口
2. 打通“我要报账”与“新建单据”流程
3. 打通“业务招待费报销”页面进入
4. 打通电子影像上传
5. 打通逐张发票识别
6. 打通单据信息填写
7. 打通明细信息填写
8. 打通保存与结果判定
9. 最后补 `task.xlsx` 双 sheet 输入、结果回写和报告

---

## 11. 当前结论

`报销单填写` 应作为当前自动化基座上的第二条业务 flow 推进。

首版不追求覆盖所有报销类型，而是：

- 以“业务招待费报销”为首个落地场景
- 使用 `task + invoice` 双 sheet 结构承接任务数据
- 先验证新 flow 能否在当前基座上跑通
- 再把其中稳定的共性能力继续沉到基座

这样既能保持架构演进节奏，也能避免一次性过度抽象。
