# Web自动化新仓库重构设计文档

## 1. 文档目的

本文档用于记录新仓库 `all-in-one_automation` 的重构思路、落地结果与后续扩展原则，目标是持续沉淀：

- 一个可复用的浏览器自动化基座
- 一个基于该基座实现的档案材料上传业务 flow
- 后续可继续扩展的新业务 flow，例如报销单自动填写

本文档不是业务 PRD，而是当前架构状态、重构结果和后续扩展边界的工程记录。

---

## 2. 重构背景

旧仓库已经具备一个可运行的 `v0.1 CLI POC`，能够完成：

- 登录外部系统
- 进入档案上传页面
- 选择人员
- 选择业务信息
- 上传文件
- 提交并确认
- 回列表页做结果校验
- 写入 SQLite、日志、截图和 `task.csv`

但旧仓库偏向“单业务专用工具”：

- 自动化基础能力和业务流程耦合较深
- `workflow.py` 承担了过多职责
- 如果后续扩展新业务 flow，复用成本较高

因此需要在新仓库中进行结构性重构，将项目演进为：

`浏览器自动化基座 + 多业务自动化 flow`

---

## 3. 新仓库定位

`all-in-one_automation` 的定位如下：

- 一个基于 Playwright 的通用 Web 自动化 CLI 平台
- 支持多个业务 flow 共用同一套浏览器自动化基座
- 支持配置化页面 selector、任务清单、批次运行、结果留痕

当前优先承载：

- 档案材料上传 flow
- 后续可扩展为报销单自动填写 flow

---

## 4. 重构原则

### 4.1 先跑通，再抽象

优先保证新架构下的材料上传主链路可跑，再逐步提高抽象层次。

### 4.2 基座按真实业务需求演进

通用浏览器自动化基座不追求一次性覆盖所有可能的自动化能力，而遵循：

- 先抽当前档案材料上传 flow 中已经真实用到、且具有复用价值的能力
- 形成第一版可复用基座
- 后续当新的业务 flow 接入时，再把其中重复出现、且值得复用的能力继续下沉到基座

一句话总结：

`基座不是一次设计完成的，而是随着业务 flow 逐轮沉淀出来的。`

### 4.3 一次只解决一层问题

避免在同一轮同时进行大规模目录调整、业务流程重写、新业务接入和测试体系重建。

### 4.4 先承接旧业务，再扩展新业务

先让档案材料上传 flow 在新基座上稳定运行，再接入报销等新业务。

---

## 5. 当前仓库结构

当前仓库已经收敛为如下主结构：

```text
all-in-one_automation/
├─ main.py
├─ README.md
├─ requirements.txt
├─ .gitignore
├─ config/
│  └─ archive_upload.yaml
├─ automation/
│  ├─ browser/
│  ├─ cli.py
│  ├─ config/
│  ├─ core/
│  ├─ runtime/
│  └─ storage/
├─ flows/
│  └─ archive_upload/
├─ samples/
├─ doc/
└─ docs/
```

目录职责：

- `automation/`：通用自动化基座，只放可复用能力
- `flows/`：具体业务 flow
- `config/`：运行配置
- `samples/`：样例任务文件
- `doc/`：中文业务与重构文档
- `docs/`：架构辅助文档

## 6. 当前重构结果

### 6.1 已完成的迁移结果

当前已经完成的关键迁移包括：

1. 浏览器自动化通用能力迁入 `automation/`
2. 档案上传业务主流程迁入 `flows/archive_upload/`
3. 主 CLI 入口迁入 `automation/cli.py`
4. `main.py` 已直接走新 CLI
5. 旧的 `dsn_uploader/` 目录已经退出当前仓库主结构

### 6.2 当前基座已承接的公共能力

当前已下沉到基座的能力包括：

- 浏览器启动和会话管理
- 页面元素点击、输入、等待
- 下拉选择
- 文件上传
- 弹窗处理
- 页面状态判断
- 批次执行和结果回写
- SQLite 留痕、日志、截图

### 6.3 当前 flow 层状态

当前档案材料上传 flow 已经实现为：

- `flows/archive_upload/flow.py`：业务主流程
- `flows/archive_upload/task_loader.py`：任务加载与校验
- `flows/archive_upload/task_model.py`：任务模型

## 7. 七轮重构计划与完成情况

七轮重构已经完成：

- 第一轮：基础动作层
- 第二轮：页面状态判断层
- 第三轮：文件上传与提交确认层
- 第四轮：人员选择与下拉选择抽象
- 第五轮：`workflow.py` 收敛为业务编排层
- 第六轮：`archive-upload` flow 迁入新 flow 层
- 第七轮：基座定型与首个通用版收尾

当前已经从“可跑”推进到“结构清晰、基座与 flow 分层明确、主链路稳定可用”的阶段。

### 第 1 轮：基础动作层

完成结果：

- locator、click、fill、visible wait、modal scope、overlay dismiss 等底层动作进入 `automation/core`
- 主流程在抽象后保持回归通过

### 第 2 轮：页面状态判断层

完成结果：

- 登录后状态判断
- 上传页状态判断
- 列表页状态判断与列表加载等待

已经进入基座能力层。

### 第 3 轮：文件上传与提交确认层

完成结果：

- 文件上传能力进入基座
- 确认提交弹窗等待与点击能力进入基座
- 档案上传主流程达到旧仓库 `v0.1` 的基本可用水平

### 第 4 轮：人员选择与下拉选择抽象

完成结果：

- 人员弹窗打开、搜索、结果等待、单选项选中、弹窗确认等能力进入基座
- 通用下拉框展开与选项点击进入基座
- `workflow.py` 中与“选人 / 选项”相关的通用逻辑明显减少

### 第 5 轮：`workflow.py` 收敛为业务编排层

完成结果：

- `workflow.py` 从“底层工具箱”收瘦为“业务编排层”
- 主要底层动作已统一走 `automation/core`
- 通用 helper、重复 wrapper 和部分兼容逻辑被清理

### 第 6 轮：`archive-upload` flow 迁入新 flow 层

完成结果：

- `flows/archive_upload/flow.py` 成为真正主实现
- 业务主流程不再依赖旧兼容实现
- 新仓库真正按“基座 + flow”结构运转，而不是“新壳包旧实现”

### 第 7 轮：基座定型与首个通用版收尾

完成结果：

- 配置、浏览器、日志、存储、任务加载与模型都已归位到新结构
- 主 CLI 入口已迁到 `automation/cli.py`
- 当前仓库已经形成“通用浏览器自动化基座 + 档案材料上传业务 flow”的第一版稳定结构

### 当前理想结构

`automation/`

- 纯通用能力

`flows/archive_upload/`

- 档案材料上传业务 flow

`automation/cli.py`

- 当前统一 CLI 入口

## 9. 当前边界与后续扩展说明

当前版本不追求：

- 一次性覆盖所有自动化能力
- 一次性接入所有未来业务 flow
- 提前设计尚未在真实业务中出现的复杂通用能力

当前版本已经实现：

- 从档案材料上传 flow 中抽出已经被真实验证的通用能力
- 形成第一版自动化基座
- 让档案材料上传 flow 成为基座上的标准样板 flow

后续基座的扩展，继续遵循：

`新业务 flow 出现 -> 识别重复能力 -> 下沉到基座`

---

## 10. 一句话总结整体计划

这次重构的核心不是“把旧代码搬到新目录”，而是：

`先搭出可复用浏览器自动化基座 -> 用档案材料上传 flow 验证基座 -> 形成稳定的基座 + flow 结构 -> 再承接更多业务 flow。`
