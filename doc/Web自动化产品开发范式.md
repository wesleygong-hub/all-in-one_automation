##### 流程范式
- 先把主链路一段一段打通
- 每次只收当前最关键的一小段
- 当前段稳定后，再推进下一阶段
- 遇到问题优先先定位根因，不盲猜、不跨阶段乱跳

##### 日志范式
- 必须有分步日志
- 每个关键动作都拆成独立子步骤
- 每条成功日志都带 elapsed_ms
- 一旦某一步不稳定，就切到诊断模式
- 诊断模式要输出足够细的信息，比如：
  - `attempts=[...]`
  - `count=...`
  - `visible=true/false`
- 当前命中的 context / frame / selector

##### 执行策略范式
- 默认走“快失败 + 快重试”
- 不做长时间死等
- 用短等待 + 高频轮询
- 优先使用最贴近人工真实操作的触发路径
- 兜底路径可以保留，但不能抢主路径、不能污染定位

##### 上下文处理范式
- iframe / page / tab 要明确跟随
- 发现真实页面结构变化时，先修 context，再修 selector
- 不把“selector 不对”和“上下文不对”混在一起判断

##### 推进原则
- 先稳定，再收轻
- 能跑通后再优化耗时
- 不为了压时间破坏稳定性
- 用户已经明确不需要修的点，不继续发散

##### 交付体验范式
- 浏览器保持打开，方便你现场核对
- 最终以真实界面结果为准，不只看日志自嗨
- 如果你已经给过 selector / DOM /页面结构，我应该优先回读现有材料，不让你重复提供
