# 报销单自动化基座能力沉淀说明

## 说明

本文用于记录 `reimbursement-fill` 在迭代过程中已经沉淀到 `automation/` 的通用基座能力，以及仍保留在 `flows/reimbursement_fill/` 内的业务能力。

本轮沉淀的目标不是把所有报销逻辑抽到基座，而是把跨业务 flow 可复用的执行、等待、上下文、诊断和失败收口能力稳定下来。报销单据类型、发票识别业务语义、字段映射、明细表格编辑等仍属于报销 flow 自身。

## 当前基座能力总览

### 1. `automation.runtime.executor`

批次执行器，负责统一编排 flow 任务生命周期。

已沉淀能力：

- `execute_batch`
  - 加载任务、校验任务、初始化浏览器与 flow 会话
  - 按任务循环执行 `flow.run_task`
  - 捕获任务异常并通过 `flow.build_failed_result` 转为失败结果
  - 写入运行记录、操作日志、任务结果回写和批次报告
  - 每条任务结束后调用 `flow.reset_task_context`
- `BatchSummary`
  - 统一批次执行摘要结构
- `Flow` 协议
  - 运行时只依赖 flow 抽象，不反向依赖具体业务 flow

适合场景：任何需要“批量任务 + 单任务执行 + 结果回写 + 失败统计”的自动化 flow。

### 2. `automation.runtime.steps`

任务步骤执行与日志基座。

已沉淀能力：

- `StepLogger`
  - 统一步骤日志回调类型
- `elapsed_ms`
  - 统一耗时输出口径
- `log_task_step`
  - 统一 `TASK + step_name + status + message` 日志格式
- `run_batch_step`
  - 封装批次级步骤的 `START / SUCCESS / elapsed_ms`
- `run_task_substep`
  - 封装任务子步骤的 `START / RETRY / SUCCESS / FAILED`
  - 默认 `retry_attempts=1`，即首次失败后快速重试 1 次
  - 默认 `retry_delay_s=0.5`
  - 支持 `non_retryable_exceptions`，明确业务异常直接抛出，不误重试
  - retry 耗尽时输出 `FAILED ... retries_exhausted=...`，然后继续抛出原异常

当前执行策略：

- 未得到明确反馈的操作，优先走快失败、快重试。
- 已明确识别的异常，交由业务 flow 声明为不可重试异常。
- runtime 只负责执行策略，不理解业务异常含义。

### 3. `automation.runtime.failures`

任务级失败收口基座。

已沉淀能力：

- `handle_task_failure`
  - 统一记录 `task_failure_guard`
  - 尝试失败截图
  - 缓存失败截图路径，默认 key 为 `_last_failure_screenshot_path`
  - 尝试执行清场逻辑
  - 截图失败或清场失败只记录对应 `FAILED` 步骤，不覆盖原始异常
  - 最终由调用方继续抛出原异常，让 executor 统一写失败结果

适合场景：任何 flow 的未预期失败兜底，包括截图、上下文清理、页面复位和失败日志收口。

### 4. `automation.core.contexts`

页面、iframe、working page 与 selector context 路由基座。

已沉淀能力：

- `selector_context_hint`
  - 从 `selector_contexts` 配置读取 selector 的上下文提示
- `append_unique_context`
  - 向候选 context 列表追加并按对象去重
- `context_debug_name`
  - 输出便于诊断的 context 名称
- `cache_page_context` / `get_cached_page_context`
  - page 级 context 缓存
- `cache_browser_context_value` / `get_cached_browser_context_value`
  - browser context 级共享值缓存
- `cache_active_working_page` / `get_cached_active_working_page`
  - 当前业务工作页缓存，自动规避已关闭页面
- `page_candidates`
  - 构造 page/main frame 候选上下文
- `resolve_selector_context`
  - 按 selector hint 和 resolver 映射解析目标上下文
- `candidate_contexts_for_selector`
  - 按 selector hint 构造候选上下文集合
- `context_matches_markers`
  - 判断 context 是否命中 marker selectors
- `get_cached_page_context_matching`
  - 读取缓存 context 并校验 marker 是否仍有效
- `resolve_first_visible_frame_context`
  - 从 iframe selector 列表中找到第一个可见 frame
- `resolve_context_by_markers`
  - 在候选 context 中按 marker 命中目标上下文

适合场景：page / iframe / 弹窗 iframe 路由、上下文缓存、多 context 候选去重和诊断命名。

### 5. `automation.core.actions`

通用页面动作基座。

已沉淀能力：

- `click_locator` / `fill_locator`
  - 基础点击与填值
- `click_locator_fast`
  - 短预算内快速重试点击
- `hover_locator_fast`
  - 短预算内快速重试 hover
- `click_latest_visible_element`
  - 点击最后一个可见元素
- `fill_locator_value`
  - 可见性检查、点击、填值和触发 `change`
- `fill_first_matching_locator`
  - 在多 context、多 selector 候选中填第一个可用控件
- `click_first_matching_locator`
  - 在多 context、多 selector 候选中点第一个可用控件
- `set_input_files`
  - 通用文件 input 设置
- `dismiss_overlays`
  - 通用浮层关闭尝试
- `upload_file` / `click_visible_upload_file_button` / `wait_uploaded_file_visible`
  - 档案上传 flow 仍在使用的通用上传辅助能力

适合场景：不带业务语义的点击、填值、hover、文件选择和多候选元素操作。

### 6. `automation.core.waits`

通用等待与状态检测基座。

已沉淀能力：

- `wait_visible`
  - 等待 selector 可见
- `wait_visible_bool`
  - 可见性快检，返回布尔值
- `ensure_visible`
  - 可见性强保证，不满足时抛明确错误
- `count_visible_elements`
  - 统计 selector 当前可见元素数量
- `locator_has_non_empty_value`
  - 判断控件或元素是否已有非空值
- `wait_markers_in_context`
  - 在单个 context 中等待任一 marker
- `wait_any_marker`
  - 在多个 context 中等待任一 marker
- `poll_until`（完整路径：`automation.core.waits.poll_until`）
  - 通用短轮询模型，收口 `while + perf_counter + wait_for_timeout`
  - 支持 timeout、interval、错误忽略和 attempts 记录
- `wait_url_contains`
  - 等待 URL 包含指定文本

适合场景：快检等待、强保证等待、跨 context marker 等待、短轮询状态等待。

### 7. `automation.core.ui_patterns`

通用 UI 交互模式基座。

已沉淀能力：

- `has_visible_dialog`
  - 判断当前 context 中是否有可见对话框
- `click_dialog_button_if_needed`
  - 在多 context 中扫描对话框并点击目标按钮
- `wait_for_condition`
  - 固定间隔等待条件成立
- `is_selected_tab_present`
  - 判断指定 tab 是否处于选中状态
- `wait_for_selected_tab_closed`
  - 等待指定选中 tab 关闭
- `wait_for_tab_closed_and_state`
  - 等待 tab 关闭并同时满足目标状态
- `click_with_fallbacks`（完整路径：`automation.core.ui_patterns.click_with_fallbacks`）
  - 统一普通点击、force 点击、JS 点击三段式 fallback
  - 已复用于弹窗按钮点击

适合场景：对话框确认、tab 关闭确认、状态等待、点击不稳定时的兜底点击。

### 8. `automation.core.diagnostics`

结构化诊断基座。

已沉淀能力：

- `AttemptLog`
  - 统一记录 attempts 诊断信息
  - 支持 `add_value(scope, name, value)`
  - 支持 `add_error(scope, name, exc)`
  - 输出统一的 `scope:name=value` 和 `scope:name=error:Type` 格式

适合场景：多 context、多 selector、多策略尝试后的失败诊断，避免 flow 中散落手写字符串拼接。

### 9. `automation.core.errors`

通用异常类型基座。

已沉淀能力：

- `AutomationError`
- `SelectorNotFoundError`
- `PageStateError`
- `VerificationError`

当前使用较轻，后续可在新 flow 中逐步替代裸 `RuntimeError`，但不强制回改已经稳定的报销 flow。

### 10. `automation.storage`

运行记录与结果回写基座。

已沉淀能力：

- `db.connect`
  - SQLite 连接管理
  - Windows 长路径/特殊路径安全处理
  - 写入可用性检查使用通用瞬时表 `__sqlite_write_check`，不会残留业务命名表
  - fallback 临时目录使用通用名称 `automation_sqlite`
- `init_db`
  - 初始化 `task_runs` 与 `operation_logs`
- `insert_task_run`
  - 写入任务运行结果
- `insert_operation_log`
  - 写入步骤日志
- `fetch_batch_summary`
  - 查询批次摘要
- `write_task_result`
  - 将任务执行结果回写到 CSV/XLSX

适合场景：所有 flow 的运行记录、操作日志、批次报告和任务文件回写。

## 报销 flow 当前接入情况

`reimbursement-fill` 已经接入的基座能力：

- 批次执行：`automation.runtime.executor.execute_batch`
- 步骤日志与 retry：`automation.runtime.steps.run_task_substep`
- 非重试异常：发票重复通过 `non_retryable_exceptions=(DuplicateInvoiceDetectedError,)` 直通
- 未预期失败收口：`automation.runtime.failures.handle_task_failure`
- context 缓存与路由：`automation.core.contexts`
- 快速点击、填值、多候选定位：`automation.core.actions`
- 可见性、marker、短轮询等待：`automation.core.waits`
- 弹窗、tab、点击 fallback：`automation.core.ui_patterns`
- attempts 诊断：`automation.core.diagnostics.AttemptLog`
- 运行记录与结果回写：`automation.storage`

报销 flow 内部半沉淀模块：

- `flows/reimbursement_fill/bill_creation.py`
- `flows/reimbursement_fill/bill_rules.py`
- `flows/reimbursement_fill/cleanup.py`
- `flows/reimbursement_fill/contexts.py`
- `flows/reimbursement_fill/image_upload.py`
- `flows/reimbursement_fill/invoice_recognition.py`
- `flows/reimbursement_fill/navigation.py`
- `flows/reimbursement_fill/page_state.py`

这些模块用于降低 `flow.py` 的复杂度，但仍属于报销业务内部能力，不属于 `automation/` 通用基座。

## 当前仍不下沉到基座的内容

以下能力继续保留在 `flows/reimbursement_fill/`：

- IAM / 财务共享 / 我要报账导航语义
- 报销单据类型、页签名称、菜单层级
- 电子影像系统语义
- 发票识别成功判断
- 重复发票业务失败判断与中止逻辑
- 报销字段映射、付款用途、业务单位、附件个数等业务字段
- 明细表格、费用分摊、业务招待、市内交通等报销领域模型
- 保存成功、回到列表、失败清场中的业务页面状态判断

原因：

- 这些能力对当前财务共享系统和报销页面强绑定。
- 不同业务 flow 或不同单据类型之间差异明显。
- 抽到 `automation/` 会把报销系统的 DOM 假设固化进基座。

## 基座边界原则

后续新增或调整能力时，按以下原则判断是否进入基座：

- 能力描述中包含明确业务名词，优先留在业务 flow。
- 能力只在当前报销页面成立，不进入基座。
- 能力可以脱离业务语义，抽象成页面级动作、等待、上下文、诊断或执行策略，才考虑进入基座。
- 能力如果只在一个 flow 中出现一次，不为了“看起来整洁”而强行下沉。
- 明确业务异常由业务 flow 定义，runtime 只提供非重试异常直通机制。
- 失败收口、截图、清场调度可以在 runtime 通用化，但具体清场动作由业务 flow 提供。

## 当前版本沉淀结论

截至本版，报销单自动化已经完成一轮较完整的“从业务 flow 反哺通用基座”沉淀。

当前基座已经具备：

- 批次执行与 flow 协议
- 步骤日志、耗时统计、快失败快重试
- 明确异常不误重试
- 任务失败 guard 与截图/清场收口
- page / iframe / working page 上下文缓存与路由
- 通用 click / fill / hover / upload 动作
- 通用可见性、marker、短轮询等待
- 通用 dialog / tab / fallback click UI 模式
- 结构化 attempts 诊断
- SQLite 运行记录、操作日志、结果回写

这一版沉淀到此收尾。后续除非接入新的业务 flow 并证明某类能力跨 flow 重复出现，否则不继续把报销业务逻辑下沉到 `automation/`。
