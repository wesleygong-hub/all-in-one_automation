# 报销单自动化基座能力沉淀说明

## 说明

本文档用于沉淀 `reimbursement-fill` 业务 flow 在迭代过程中已经验证可复用的基座能力，并明确哪些能力已经下沉到 `automation/`，哪些能力仍保留在业务 flow 中。

目标不是把所有业务逻辑都抽到基座，而是优先沉淀：

- 跨业务场景都会重复出现的页面/iframe/context 能力
- 通用步骤执行、日志、重试能力
- 通用 UI 交互模式
- 通用动作与等待能力

## 已下沉的基座能力

### 1. `automation.runtime.steps`

用于统一任务步骤执行、日志输出、耗时统计与重试语义。

已沉淀能力：

- `StepLogger`
  用于统一步骤日志回调类型，约束业务 flow 在记录步骤日志时的入参形式。
  适合场景：任务执行器、flow 内部步骤封装、需要把步骤日志同时写入控制台与结果存储时。

- `elapsed_ms`
  用于根据起始时间计算步骤耗时，统一 `elapsed_ms` 输出口径。
  适合场景：所有需要输出耗时的批次级步骤、任务级步骤、诊断日志。

- `log_task_step`
  用于输出单个任务步骤日志，统一 `TASK + step_name + status + message` 的日志格式。
  适合场景：普通任务步骤开始、成功、失败、跳过等状态记录。

- `run_batch_step`
  用于封装批次级步骤执行，统一 `START / SUCCESS / FAILED` 与耗时日志。
  适合场景：登录、进入业务首页、进入列表页等“整批任务只做一次”的首部动作。

- `run_task_substep`
  用于封装任务级子步骤执行，统一日志、耗时与 `500ms` retry 机制。
  适合场景：点击按钮、检测页面到达、填写字段、保存单据等每条任务都会重复执行的步骤。

适用场景：

- 所有业务 flow 的 `START / SUCCESS / RETRY / FAILED / SKIP`
- 统一 `elapsed_ms`
- 统一 `500ms` retry 机制

### 2. `automation.core.contexts`

用于统一 selector 的上下文提示、iframe/page context 解析、缓存与候选上下文管理。

已沉淀能力：

- `selector_context_hint`
  用于从 `selector_contexts` 配置中读取 selector 对应的上下文提示。
  适合场景：一个 selector 明确属于 `page`、业务 iframe、影像 iframe、弹窗 iframe 等固定上下文时。

- `append_unique_context`
  用于向候选上下文列表中追加 context，并自动去重。
  适合场景：组合多个 page/frame 候选上下文，避免重复扫描同一个 context。

- `context_debug_name`
  用于把 page 或 frame 转成便于日志输出的上下文名称。
  适合场景：诊断 `attempts=[...]`、多 iframe 扫描日志、错误定位输出。

- `cache_page_context`
  用于把某个 page 级上下文缓存到 page 对象上。
  适合场景：已解析到某个稳定 iframe，后续同一任务中反复复用。

- `get_cached_page_context`
  用于读取 page 上已缓存的上下文。
  适合场景：避免重复解析同一业务 iframe、表单 iframe、影像 iframe。

- `cache_browser_context_value`
  用于把值缓存在浏览器上下文级别，而不是单个 page 级别。
  适合场景：跨标签页复用状态、复用活动工作页、批次级共享值。

- `get_cached_browser_context_value`
  用于读取浏览器上下文级别缓存值。
  适合场景：多标签页切换后继续复用之前的活动页或全局状态。

- `cache_active_working_page`
  用于缓存“当前真正执行业务操作的工作页”。
  适合场景：一个批次中打开了新 tab，后续任务需要继续复用该 tab，而不是回到初始 page。

- `get_cached_active_working_page`
  用于获取当前缓存的活动工作页，并自动规避已关闭 page。
  适合场景：下一条任务开始时直接复用“我要报账”所在页、复用当前业务主工作页。

- `resolve_selector_context`
  用于根据 selector 的 context hint 和 resolver 映射，直接解析该 selector 所属上下文。
  适合场景：明确知道某个 selector 属于哪层 iframe，希望避免全量扫描时。

- `candidate_contexts_for_selector`
  用于按 selector 的 context hint 构造候选上下文集合。
  适合场景：先按配置约束缩小范围，再在少量候选 context 中尝试元素定位。

- `context_matches_markers`
  用于判断某个 context 是否命中一组 marker selector。
  适合场景：检测当前 iframe 是否为目标页面、检测缓存 context 是否仍然有效。

- `get_cached_page_context_matching`
  用于读取缓存 context，并进一步校验它是否仍命中 marker。
  适合场景：页面切换后避免误复用失效 iframe。

- `resolve_first_visible_frame_context`
  用于在一组 iframe selector 中找到第一个可见 iframe，并返回其 frame context。
  适合场景：目标 iframe 结构相对稳定，但 selector 可能有多个候选时。

- `resolve_context_by_markers`
  用于在候选 context 列表中，找到第一个命中 marker 的 context。
  适合场景：无法直接靠 URL 或单个 selector 确认 context 时，通过页面特征识别目标 iframe。

适用场景：

- page / iframe / 外层 iframe / 弹窗 iframe 路由
- selector 所属上下文配置化
- context cache 复用
- 多 context 候选去重与诊断命名

### 3. `automation.core.ui_patterns`

用于统一通用 UI 交互模式，例如弹框确认、tab 关闭检测、关闭后返回目标状态检测。

已沉淀能力：

- `has_visible_dialog`
  用于快速判断当前 context 中是否出现可见对话框。
  适合场景：点击关闭、删除、提交后，先判断是否弹出了确认框。

- `click_dialog_button_if_needed`
  用于在多个候选 context 中扫描确认框按钮，并尝试点击目标按钮文本。
  适合场景：确认框可能出现在 page、业务 iframe、外层 iframe 中的场景。

- `wait_for_condition`
  用于按固定轮询间隔等待一个条件成立。
  适合场景：等待页面状态变化、等待 tab 关闭、等待 marker 消失或出现。

- `is_selected_tab_present`
  用于判断当前是否仍存在某个被选中的 tab。
  适合场景：判断“电子影像”“业务招待费报销”等页签是否仍在当前激活状态。

- `wait_for_selected_tab_closed`
  用于等待某个选中 tab 关闭。
  适合场景：关闭子页面后，确认对应 tab 已经消失。

- `wait_for_tab_closed_and_state`
  用于等待 tab 关闭，并同时满足某个目标页面状态。
  适合场景：关闭报销单页签后，需要确认已经返回我要报账列表。

适用场景：

- 确认弹框点击“确定/取消”
- 检测 tab 是否已关闭
- 关闭 tab 后是否返回目标页面

### 4. `automation.core.actions`

用于统一高频、通用的页面动作。

已沉淀能力：

- `click_locator_fast`
  用于在较短预算内快速重试点击某个 selector。
  适合场景：按钮已接近可用，只需要“快失败、快重试”的点击动作。

- `click_latest_visible_element`
  用于在一组同类元素里，点击最后一个可见元素。
  适合场景：动态新增面板、下拉项、最后一个弹出的箭头或按钮优先作为操作目标时。

- `fill_locator_value`
  用于对单个输入控件执行标准填值动作，包括可见性检查、点击、填值与触发 `change`。
  适合场景：普通输入框、textarea、可直接 `fill()` 的表单控件。

- `fill_first_matching_locator`
  用于在多个 context 和多个 selector 候选中，找到第一个可用输入控件并填值。
  适合场景：同一字段可能出现在多个 iframe 中，或同一字段存在多条 selector 兜底时。

适用场景：

- 快速点击可见控件
- 点击最后一个可见元素
- 单个输入控件通用填值
- 多上下文、多 selector 候选下找到第一个可填控件并填值

### 5. `automation.core.waits`

用于统一高频等待与元素状态检测。

已沉淀能力：

- `wait_visible_bool`
  用于返回元素是否在给定时间内变为可见，而不是直接抛错。
  适合场景：轻量快检、页面到达预检、诊断扫描。

- `ensure_visible`
  用于保证元素在给定时间内可见，否则抛出明确错误。
  适合场景：业务动作执行前的前置校验，例如按钮、输入框、下拉箭头必须可见。

- `count_visible_elements`
  用于统计某个 selector 当前可见元素数量。
  适合场景：检测上传缩略图数量、候选按钮数量、可见 tab 数量、明细行数量。

- `locator_has_non_empty_value`
  用于判断输入控件或元素是否已经具有非空值。
  适合场景：识别完成后判断发票号码是否回填、选择完成后判断控件值是否真正写入。

适用场景：

- 可见性快检
- 保证元素可见
- 可见元素计数
- 输入框/控件是否已有值

## 当前仍保留在业务 flow 的能力

以下能力仍建议保留在 `flows/reimbursement_fill/flow.py`，不建议继续强行下沉：

### 1. 报销业务语义

- 业务招待费报销单据的进入路径
- 重复发票判定后的中止逻辑
- 保存单据后返回我要报账列表

原因：

- 已明显带业务语义
- 不同业务系统和单据类型差异较大

### 2. 报销单字段语义

- 业务单位
- 付款用途
- 接待类型
- 公司人数
- 核定报账金额
- 备注

原因：

- 字段名、录入顺序、录入规则都强业务绑定

### 3. 明细表格编辑机制

- 明细行重建
- 默认行删除
- 新增明细行
- 表格编辑态切换
- 逐列填值与组合选择

原因：

- 虽然“表格编辑”本身有抽象空间，但当前版本仍然高度依赖该报销页面的具体 DOM 与交互特征

## 后续建议

### 建议继续复用基座的方向

- 新业务 flow 优先复用 `steps`
- selector 统一配置 `selector_contexts`
- 新页面交互优先复用 `actions / waits / ui_patterns`
- 新 flow 内优先保留业务语义函数，减少一开始过度抽象

### 不建议当前阶段继续强抽的方向

- 识别成功、重复发票、保存成功等业务结果判定
- 报销明细表格编辑引擎
- 财务共享页面特有 selector
- 单据字段业务映射

## 当前阶段结论

`reimbursement-fill` 已经完成一轮比较完整的“从业务 flow 反哺基座”的沉淀。

当前基座已经具备：

- 通用步骤执行与日志能力
- 通用 context 路由与缓存能力
- 通用弹框 / tab / 等待模式
- 通用点击 / 填值 / 可见性检测能力

后续再接入新的业务 flow 时，应优先验证：

- 是否能直接复用这些基座能力
- 只有在多个业务 flow 反复出现后，再继续推进下一轮下沉

## 基座能力边界定义

为避免后续迭代过程中再次把业务逻辑混入 `automation/`，当前阶段先明确基座能力边界如下。

### 1. 明确属于基座的能力

以下能力已经确认属于基座，应继续放在 `automation/` 下统一维护：

- 浏览器会话管理
- 配置加载、环境变量展开、运行目录准备
- page / iframe / working page 的上下文解析、缓存、命中与回退
- 通用 click / fill / select / upload 等页面动作
- 通用对话框、tab、等待、关闭确认模式
- 批次执行、步骤日志、任务结果回写、失败截图、批次报告
- CLI 统一入口与导入自检能力

这些能力的共同特征是：

- 不依赖“报销单”“电子影像”“费用分摊”等业务语义
- 在不同业务 flow 中具备重复使用价值
- 即使未来切换业务系统，抽象层仍然成立

### 2. 暂不进入基座的能力

以下能力当前不建议继续下沉到 `automation/`，应保留在 `flows/reimbursement_fill/flow.py` 或报销 flow 自己的模块中：

- 电子影像业务语义
- 发票识别与重复发票业务判定
- 报销单页签语义与切换顺序
- 费用分摊 / 报销明细字段逻辑
- 财务共享页面特有 iframe 结构适配
- 报销单保存成功、回列表、失败清场等业务收尾逻辑

这些能力的共同特征是：

- 对当前财务共享系统和报销页面高度绑定
- 不同单据类型之间也存在明显差异
- 即使抽出来，也很难在其他 flow 中直接复用

### 3. 当前阶段的边界原则

后续新增或修改能力时，统一按以下原则判断是否进入基座：

- 如果能力描述中已经包含明显业务术语，优先留在业务 flow
- 如果能力只是在“当前报销页面”成立，不进入基座
- 如果能力可以脱离业务术语、抽象成页面级通用动作或等待，再考虑进入基座
- 如果某能力仅在一个 flow 中出现一次，不为了“看起来整洁”而强行下沉

### 4. 面向后续 flow 的使用原则

后续新增业务 flow 时，优先遵循以下口径：

- 能复用 `steps` 的，优先复用 `steps`
- 能复用 `contexts` 的，优先复用 `contexts`
- 能复用 `actions / waits / ui_patterns` 的，优先复用已有页面动作和模式
- 业务 flow 只负责业务编排，不重新发明通用页面能力

## 半沉淀层后续规范补充项

当前仓库里还有一部分能力已经出现了复用趋势，但尚未沉淀成统一规则。下一阶段不急于抽业务组件，而是优先补这部分“规范层”。

### 1. 等待模型规范

当前等待逻辑分散在：

- `automation.core.actions`
- `automation.core.waits`
- `automation.core.ui_patterns`
- `flows/reimbursement_fill/flow.py`

下一步建议补的不是新组件，而是统一等待语义：

- 什么叫“快检等待”
- 什么叫“强保证等待”
- 什么叫“状态等待”
- 什么情况下允许 retry
- 什么情况下应直接 fail

目标是避免不同 flow 再各自长出一套等待风格。

建议先统一为以下三类等待模型：

#### 1.1 快检等待

定义：

- 用较短时间快速判断“元素/状态是否已经就绪”
- 不要求一定成功
- 更适合作为分支判断或兜底前预检

适用场景：

- 快速判断某个按钮是否已经出现
- 快速判断某个 tab 是否已经切中
- 快速判断某个 iframe marker 是否已经可见
- 诊断扫描时统计可见元素数量

建议特征：

- 超时时间短
- 失败返回 `False` 或空结果
- 不直接抛出业务异常

当前可归入快检等待的能力：

- `wait_visible_bool`
- `has_visible_dialog`
- `locator_has_non_empty_value`
- 部分 context marker 命中检测

#### 1.2 强保证等待

定义：

- 某个元素或页面状态是后续动作的硬前置条件
- 如果在预算内未满足，应直接失败

适用场景：

- 点击前要求按钮必须可见
- 填值前要求输入框必须可编辑
- 上传前要求上传控件必须就绪
- 保存前要求保存按钮已可操作

建议特征：

- 失败直接抛错
- 错误信息要明确指出缺失的前置条件
- 用于串行主流程中的关键步骤

当前可归入强保证等待的能力：

- `ensure_visible`
- `fill_locator_value` 中的前置可见性保证
- 各类 `wait_for(... state="visible")`

#### 1.3 状态等待

定义：

- 等待页面进入某种业务状态，而不只是某个元素可见
- 适合“动作已执行，等待结果收敛”的场景

适用场景：

- 等待 tab 关闭
- 等待关闭后回到目标页面
- 等待识别结果出现
- 等待弹框关闭
- 等待页面从一个业务阶段切换到另一个业务阶段

建议特征：

- 以条件轮询为主
- 允许由多个信号共同判定
- 更关注“页面是否进入目标状态”，而不是单点元素

当前可归入状态等待的能力：

- `wait_for_condition`
- `wait_for_selected_tab_closed`
- `wait_for_tab_closed_and_state`
- flow 中各类“页面到达 / 页面关闭 / 状态切换”检测

#### 1.4 使用原则

后续新增等待逻辑时，建议统一按以下顺序判断：

1. 这是快检，还是强保证，还是状态等待
2. 如果只是分支判断，优先使用快检等待
3. 如果是主流程硬前置条件，使用强保证等待
4. 如果是动作后的结果收敛，使用状态等待

额外约束：

- 不建议把“快检等待”直接当成“主流程完成判定”
- 不建议在业务 flow 中反复手写裸 `wait_for_timeout` 代替状态等待
- 不建议在同一类场景里混用多套等待口径

#### 1.5 后续落地建议

后续代码整理时，可以按这三类等待模型继续收口：

- `waits.py` 负责元素级等待
- `ui_patterns.py` 负责页面交互模式等待
- `flow.py` 只保留业务状态判定，不再继续沉积底层等待细节

#### 1.6 当前代码示例

当前仓库里已经有较典型的等待模型对应关系：

- 快检等待示例  
  `automation.core.waits.wait_visible_bool`  
  用于快速判断元素是否在短时间内可见，适合作为预检或分支判断。

- 强保证等待示例  
  `automation.core.actions.fill_locator_value`  
  在填值前先保证目标控件可见，不满足则直接失败。

- 状态等待示例  
  `automation.core.ui_patterns.wait_for_tab_closed_and_state`  
  用于等待 tab 关闭并同时满足目标页面状态，属于典型“动作后等待结果收敛”的模式。

### 2. selector / selector_context 规范

当前 selector 与 context hint 机制已经具备，但仍需要补统一规则：

- selector 命名规范
- `selector_contexts` 的推荐使用方式
- 哪些 selector 必须显式配置上下文
- page 级 selector 与 iframe 级 selector 的边界
- 业务 selector 与通用 selector 的边界

目标是避免 selector 体系继续松散增长。

建议统一以下规范：

#### 2.1 selector 命名规则

selector 命名建议采用“业务语义 + 控件类型/动作”的方式，保证可读性与可维护性。

推荐形式：

- `iam_username_input`
- `finance_share_entry`
- `electronic_image_tab_entry`
- `detail_add_button`
- `reimbursement_bill_close_confirm_button`

命名原则：

- 名称应尽量表达“这个元素在业务上是什么”
- 避免用纯 DOM 结构命名，例如 `left_button_1`
- 避免把过多实现细节写进名字，例如 `span_input_arrow_div`

#### 2.2 selector_contexts 的使用规则

`selector_contexts` 用于表达“这个 selector 理论上属于哪类上下文”，它的作用是：

- 缩小查找范围
- 降低全量扫描成本
- 让 context 解析逻辑配置化

推荐适用场景：

- selector 稳定属于某类 iframe
- selector 在 page 层和 iframe 层容易混淆
- selector 对性能敏感，需要优先缩小上下文范围

不建议适用场景：

- selector 所属上下文本身极不稳定
- 当前还没有充分验证 context 规律
- 只是为了“看起来规范”而提前配置

#### 2.3 哪些 selector 应优先显式配置 context

优先建议显式配置 `selector_contexts` 的 selector 包括：

- 业务主按钮
- 影像系统按钮
- 表单输入控件
- 明细表格操作按钮
- 确认弹框按钮

原因是这些控件：

- 往往操作频繁
- 对性能和稳定性都敏感
- 最容易在多 iframe 页面中被误命中

#### 2.4 page 级 selector 与 iframe 级 selector 边界

建议按以下方式区分：

- 顶层导航、菜单、系统入口类控件：优先归 `page`
- 单据表单区、影像区、弹窗区控件：优先归对应 iframe
- 如果同名元素在顶层和 iframe 中都可能出现，优先显式配置 `selector_contexts`

目标是避免：

- 明明应在单据 iframe 内点击，却误在 page 上扫描
- 明明只需要少量上下文，却退化成全页面搜索

#### 2.5 通用 selector 与业务 selector 边界

建议继续保持以下边界：

- 通用 selector 机制放在基座
- 具体 selector 实例值放在各 flow 配置文件

也就是说，基座负责：

- selector 查找
- selector 所属上下文路由
- selector 诊断输出

业务 flow 负责：

- 某个字段、按钮、tab 的具体 selector 写法

这样可以避免把具体业务系统 DOM 写死在 `automation/` 中。

#### 2.6 当前代码示例

当前仓库里已经有较明确的 selector/context 分层示例：

- 通用 selector 路由机制示例  
  `automation.core.contexts.resolve_selector_context`  
  基座只负责根据 `selector_contexts` 把 selector 路由到正确上下文。

- 业务 selector 配置示例  
  `config/reimbursement_fill.yaml` 中的：
  - `electronic_image_tab_entry`
  - `detail_add_button`
  - `reimbursement_bill_close_confirm_button`

- selector_contexts 配置示例  
  `config/reimbursement_fill.yaml` 中的：
  - `electronic_image_tab_entry: "bill_iframe"`
  - `detail_add_button: "bill_form_iframe"`
  - `reimbursement_bill_close_confirm_button: "bill_outer_iframe"`

这说明当前机制已经具备，只需要继续补规则，而不是推翻重来。

### 3. 失败处理分层规范

当前失败处理能力已经存在，但还缺少统一语义层。后续建议补齐：

- 异常失败
- 业务失败
- 可恢复失败
- 是否截图
- 是否清场
- 是否写回任务文件
- 是否进入批次失败统计

目标是让不同 flow 的失败处理口径一致，而不是靠各自 `RuntimeError` 文本临时约定。

建议统一为以下三类失败：

#### 3.1 异常失败

定义：

- 代码执行过程中发生非预期异常
- 主流程无法继续
- 通常需要截图、清场、写回失败结果

典型场景：

- 找不到关键 selector
- context 解析异常
- 页面状态与预期严重不符
- Playwright 动作直接抛异常且无法恢复

建议处理方式：

- 记录异常堆栈
- 截取失败现场
- 执行失败清场
- 写回任务结果
- 计入批次失败统计

#### 3.2 业务失败

定义：

- 程序本身没有崩，但业务规则判断该任务应失败
- 属于“流程正常结束，但结果是 failed”

典型场景：

- 重复发票
- 无法填单
- 识别结果不满足业务规则
- 当前记录不允许继续提交

建议处理方式：

- 不一定需要异常堆栈
- 优先记录清晰业务原因
- 可在返回失败前先截图
- 执行清场
- 写回任务结果
- 计入批次失败统计

#### 3.3 可恢复失败

定义：

- 当前动作失败，但不代表任务整体失败
- 允许通过 retry、fallback context、兜底 selector 等方式继续推进

典型场景：

- 单次点击失败
- 单次等待超时
- 主 context 未命中，fallback context 可继续尝试
- 某个 selector 失效，但备用 selector 可命中

建议处理方式：

- 不直接终止任务
- 优先在当前步骤内完成 retry / fallback
- 只在恢复失败后升级为异常失败或业务失败

#### 3.4 失败处理统一决策项

后续所有 flow 的失败处理建议至少统一判断这几个问题：

- 这是异常失败、业务失败，还是可恢复失败
- 是否需要截图
- 是否需要清场
- 是否需要写回任务文件
- 是否需要写入 SQLite
- 是否需要进入批次失败统计

#### 3.5 当前版本的落地口径

结合当前实现，建议默认口径如下：

- 异常失败：截图 + 清场 + 回写 + 入库 + 计入失败
- 业务失败：截图 + 清场 + 回写 + 入库 + 计入失败
- 可恢复失败：优先在步骤内 retry / fallback，不立刻终止任务

这样可以保证：

- 真正失败的任务都能留存现场
- 主流程不会因为一次短暂抖动就直接失败
- 日志、结果回写和批次汇总口径保持一致

#### 3.6 当前代码示例

当前仓库里已经有较典型的失败处理分层示例：

- 异常失败示例  
  `automation.runtime.executor.execute_batch`  
  在 `flow.run_task(...)` 抛异常后，统一记录异常、兜底生成失败结果，并继续走截图、入库、回写逻辑。

- 业务失败示例  
  `flows.reimbursement_fill.flow._abort_duplicate_invoice_task(...)`  
  重复发票场景不是程序崩溃，而是业务上明确失败，并在返回失败前先截图、再清场。

- 可恢复失败示例  
  `automation.runtime.steps.run_task_substep`  
  单步骤失败后先做一次 `500ms` retry，而不是立刻终止整个任务。

## 建议优先落代码项

文档层面的边界和规范已经基本清楚，下一阶段建议尽快开始落代码，但不直接大拆业务 flow，而是优先做“低风险、高复用”的规范化收口。

建议优先级如下：

### 1. 优先落地：等待模型收口

目标：

- 让 `waits.py / ui_patterns.py / actions.py` 的职责更清晰
- 让 flow 中零散的等待逻辑继续回收到基座

建议先做的代码动作：

- 梳理 `wait_visible_bool / ensure_visible / wait_for_condition` 的使用边界
- 把 flow 中重复出现的短轮询状态等待，优先归并到 `ui_patterns.py`
- 减少 flow 中裸 `wait_for_timeout` 的直接使用

### 2. 第二优先：selector_context 口径收口

目标：

- 让 selector 与 context 的关系更稳定、更可配置

建议先做的代码动作：

- 统一 `selector_contexts` 的命名口径
- 盘点 `reimbursement_fill.yaml` 中哪些 selector 已配置 context，哪些仍应补配置
- 对高频按钮、输入框、确认按钮优先补齐 context hint

### 3. 第三优先：失败处理口径收口

目标：

- 让异常失败、业务失败、可恢复失败在代码里更容易区分

建议先做的代码动作：

- 在 flow 中减少直接抛出语义模糊的 `RuntimeError`
- 对明显的业务失败场景优先形成统一返回路径
- 对可恢复失败场景继续下沉到步骤级 retry / fallback

### 4. 当前阶段明确不做的事项

为了避免过度设计，当前阶段先明确以下事项不推进：

- 不抽“报销共享组件”
- 不做通用“财务共享组件库”
- 不把电子影像、费用分摊、报销明细等强业务能力下沉到基座

原因是这些能力当前仍具有明显的页面定制和业务定制属性，过早抽象会削弱基座边界，反而增加维护成本。

## 当前建议

如果马上开始落代码，最适合先做的是：

- 先收口等待模型
- 再收口 selector_context

原因是这两项：

- 对两个 flow 都有复用价值
- 风险低于直接抽业务组件
- 能立刻为下一轮业务扩展提供稳定收益
