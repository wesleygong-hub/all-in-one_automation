| 项目 | 内容 |

### 1.1 产品一句话定义

一个基于浏览器自动化的 CLI 工具，用于模拟人工登录外部资料管理系统，批量完成档案材料上传、登记、校验与操作留痕。

### 1.2 编写背景

当前团队需要将本地已整理好的档案材料批量上传至外部资料管理系统。该系统暂无直接可调用的正式开放接口，当前可行路径是通过浏览器访问前端页面，并基于已授权的登录凭据和访问密钥，模拟人工操作完成上传和登记流程。

为了尽快验证业务链路可行性，第一阶段先做纯 CLI 的 POC 版本，不做 Web 页面和桌面 GUI；后续在 CLI 核心能力稳定后，再演进为更易用的管理界面。

---

## 2. 项目背景与问题

### 2.1 当前现状

- 操作人员需要手工登录外部系统
- 按任务逐条检索目标记录
- 手工上传本地档案文件
- 手工填写相关登记字段
- 手工确认结果并记录执行情况

### 2.2 当前痛点

- 重复劳动多，处理效率低
- 人工上传容易出错，存在漏传、错传、重复传风险
- 缺少统一的日志、截图和审计记录
- 批量处理时难以断点续传和失败重试
- 出现异常时难以快速定位失败步骤

### 2.3 建设目标

通过一个可执行的 CLI 自动化工具，先打通最小业务闭环：

- 自动登录
- 自动查询目标记录
- 自动上传档案材料
- 自动填写登记字段
- 自动提交和结果校验
- 自动记录日志和执行结果

---

## 3. 产品目标

### 3.1 核心目标

1. 让操作人员可以通过命令行一次性发起一批上传任务。
2. 程序能够模拟人工在浏览器中完成上传和登记操作。
3. 任务执行全程有日志、有截图、有结果状态可追踪。
4. 单条任务失败不阻断批次，失败后可继续下一条。
5. 核心能力与 UI 解耦，为后续 Web/GUI 版本打基础。

### 3.2 非目标

本阶段不做以下内容：

- 不直接对接外部系统后端接口
- 不绕过现有权限、登录或安全控制
- 不建设多用户协作后台
- 不做复杂审批流、权限系统、消息系统
- 不做 OCR 智能识别或材料自动分类平台

## 4. 用户画像与使用场景

### 4.1 目标用户

- 档案材料上传专员
- 业务支持人员
- 技术实施/运维人员

### 4.2 核心使用场景

#### 场景 A：批量上传档案材料

操作人员准备好任务清单和本地文件，执行 CLI 命令，系统自动登录外部平台并完成批量上传与登记。

#### 场景 B：失败任务补跑

部分任务因页面异常、网络波动、字段不匹配等失败后，操作人员修正问题并重新运行，程序只重试失败任务。

#### 场景 C：异常留痕后继续

当系统遇到页面异常、选择器失效、字段不匹配或无法自动判定结果时，程序记录失败原因、保存截图，并继续执行下一条任务。

---

## 5. 业务边界与原则

### 5.1 业务边界

- 输入：本地整理好的任务清单和材料文件
- 处理：自动化登录、查询、上传、登记、校验
- 输出：执行结果、失败原因、截图、日志、报表

### 5.2 设计原则

- 以模拟人工浏览器操作为主
- 配置优先，避免大量硬编码
- 模块解耦，便于后续扩展
- 先打通闭环，再逐步增强
- v0.1 优先保证失败留痕和批次连续执行，不引入人工确认后继续的交互流程

---

## 6. 总体方案

### 6.1 方案概述

系统读取配置文件和任务清单后，启动浏览器，自动登录外部资料管理系统，逐条查询目标记录，并完成文件上传、元数据填写、提交确认和结果校验；所有步骤均写入日志和本地数据库，最终输出批次执行报告。

### 6.2 版本路线

#### Phase 1：CLI POC

聚焦最小可行链路：

- CLI 命令启动
- 配置管理
- 单账号登录
- CSV/Excel 任务导入
- 批量上传与登记
- 日志、截图、SQLite 记录
- 失败重试和断点续传

#### Phase 2：Web / 桌面 GUI

在不重写核心引擎的情况下扩展：

- 可视化任务配置
- 实时进度面板
- 历史批次查询
- 失败任务一键重跑
- 多账号、多模板管理

## 6.3 v0.1 绝对范围

为避免 POC 阶段范围持续膨胀，v0.1 版本采用“必须做 / 明确不做”的硬边界管理。开发过程中如遇新增想法，默认归入后续版本，除非它直接影响当前最小闭环。

### 6.3.1 v0.1 必须做

#### CLI 能力

- `validate`
- `run`
- `report`
- `--config`
- `--tasks`
- `--headed`

#### 浏览器自动化主流程

- 打开登录页
- 用户名 + 密码登录
- 进入上传档案附件页面
- 选择人员
- 选择业务类型
- 上传文件
- 提交表单
- 自动跳转回列表页
- 读取列表第一页第一条记录
- 按 `employee_id + business_type` 判定上传是否成功

#### 数据与留痕

- 终端实时输出执行过程
- SQLite 记录任务结果和步骤日志
- 失败时截图
- 生成批次执行报告

#### 输入与校验

- 支持读取 `task.csv`
- 按当前已确认的 6 个字段处理任务
- v0.1 对“校验本地文件”的范围仅为校验必填项是否已填写

### 6.3.2 v0.1 明确不做

以下内容不纳入 v0.1，避免 POC 范围失控：

- `--headless`
- 失败任务重跑
- 断点续传增强能力
- 复杂文件校验
- 文件存在性、大小、格式深度校验
- 多账号支持
- 多任务筛选参数
- 单任务执行参数
- 行号区间执行参数
- 静默模式与高级调试模式
- Web 界面
- 桌面 GUI

### 6.3.3 v0.1 成功标准

只要满足以下条件，即认为 v0.1 达成目标：

- 用户可以通过 CLI 启动程序
- 程序可以读取配置与任务清单
- 程序可以自动打开浏览器并完成登录
- 程序可以进入上传页并完成一次完整上传流程
- 程序可以在提交后回到列表页
- 程序可以读取第一条记录并按 `employee_id + business_type` 判定成功与否
- 程序可以把过程展示在终端，并把结果写入 SQLite
- 程序在失败时能够截图并记录错误

### 6.3.4 范围控制原则

- 只为“打通端到端闭环”服务的需求进入 v0.1
- 凡是“提高效率但不影响闭环”的能力，默认放入后续版本
- 凡是“增强稳定性但会显著增加开发量”的能力，优先作为 v0.2 迭代项
- 在开发中如需新增功能，先判断是否影响最小闭环；不影响则不进入 v0.1

## 7. CLI POC 功能需求

## 7.1 配置管理

系统需支持通过配置文件统一管理运行参数。

配置项至少包括：

- 外部系统访问地址
- 登录用户名
- 密码或授权密钥
- 浏览器运行模式
- 页面选择器配置
- 字段映射配置
- 重试次数和超时时间
- 日志路径、截图路径、数据库路径

需求要求：

- 支持 YAML 配置文件
- 支持敏感参数通过环境变量注入
- 支持不同环境配置切换
- 程序启动时对配置合法性进行校验

---

### 7.1.1 v0.1 配置文件最终结构

为避免 POC 阶段配置项过多，v0.1 建议仅保留以下 6 组配置：`system`、`auth`、`paths`、`selectors`、`mapping`、`runtime`。

#### `system`

用于描述目标系统地址和浏览器基础运行参数。

| 配置项 | 是否必填 | 含义 | v0.1 用途 |
|------|----------|------|-----------|
| `base_url` | 是 | 外部系统登录页地址 | 程序启动后首先访问该地址 |
| `timeout_ms` | 是 | 默认超时时间，单位毫秒 | 页面加载、元素等待、页面跳转超时控制 |
| `headed` | 是 | 是否以有界面方式启动浏览器 | POC 默认建议为 `true`，方便观察和人工介入 |

#### `auth`

用于描述登录凭据。

| 配置项 | 是否必填 | 含义 | v0.1 用途 |
|------|----------|------|-----------|
| `username` | 是 | 登录用户名 | 自动填写登录页用户名 |
| `password` | 是 | 登录密码 | 自动填写登录页密码 |

说明：

- 当前已确认登录流程仅需要“用户名 + 密码”
- 若后续页面确实需要授权密钥，再在后续版本增加 `access_key`
- 密码类配置建议支持环境变量注入

#### `paths`

用于描述本地输出与存储位置。

默认规则：

- 若用户未显式指定绝对路径，则所有输出与存储位置默认相对于当前项目根目录
- 程序启动或首次写入时，应自动创建所需子文件夹
- v0.1 建议统一在项目根目录下使用 `data/` 与 `output/` 两类子目录

| 配置项 | 是否必填 | 含义 | v0.1 用途 |
|------|----------|------|-----------|
| `sqlite_path` | 是 | SQLite 数据库文件路径，默认位于当前项目路径下的 `data/` 子目录 | 存储任务结果和步骤日志 |
| `screenshot_dir` | 是 | 失败截图输出目录，默认位于当前项目路径下的 `output/screenshots/` 子目录 | 出错时自动截图 |
| `report_dir` | 是 | 批次报告输出目录，默认位于当前项目路径下的 `output/reports/` 子目录 | 保存执行汇总结果 |

#### `selectors`

用于描述浏览器自动化中的页面元素定位方式，是 v0.1 核心配置。

##### 登录页

| 配置项 | 是否必填 | 含义 |
|------|----------|------|
| `username_input` | 是 | 用户名输入框选择器 |
| `password_input` | 是 | 密码输入框选择器 |
| `login_button` | 是 | 登录按钮选择器 |
| `privacy_agreement_checkbox` | 视情况 | “同意并接受用户隐私协议”复选框选择器 |

##### 首页 / 入口页

| 配置项 | 是否必填 | 含义 |
|------|----------|------|
| `upload_entry` | 是 | “上传档案附件”入口按钮或菜单选择器 |

##### 上传页

| 配置项 | 是否必填 | 含义 |
|------|----------|------|
| `person_select_button` | 是 | 打开“选择人员”弹窗的按钮 |
| `business_line_select` | 是 | 所属业务控件选择器 |
| `business_type_select` | 是 | 业务类型控件选择器 |
| `upload_file_button` | 是 | 上传文件按钮或文件输入控件 |
| `submit_button` | 是 | 提交按钮选择器 |

##### 人员选择弹窗

| 配置项 | 是否必填 | 含义 |
|------|----------|------|
| `person_query_input` | 是 | 人员查询输入框 |
| `person_query_button` | 是 | 人员查询按钮 |
| `person_result_first_row` | 是 | 查询结果第一条记录 |
| `person_confirm_button` | 是 | 人员确认按钮 |

##### 提交确认框

| 配置项 | 是否必填 | 含义 |
|------|----------|------|
| `confirm_ok_button` | 是 | 提交确认弹窗中的“确定”按钮 |

##### 列表校验页

| 配置项 | 是否必填 | 含义 |
|------|----------|------|
| `list_first_row` | 是 | 列表第一页第一条记录 |
| `list_first_row_employee_id` | 是 | 第一条记录中的 `employee_id` 字段 |
| `list_first_row_business_type` | 是 | 第一条记录中的 `business_type` 字段 |

#### `mapping`

用于描述任务清单值与页面显示值之间的映射关系。

| 配置项 | 是否必填 | 含义 | v0.1 用途 |
|------|----------|------|-----------|
| `business_line` | 否 | `business_line` 映射表 | 当任务值与页面展示值不一致时做转换 |
| `business_type` | 否 | `business_type` 映射表 | 当任务值与页面展示值不一致时做转换 |

说明：

- 若任务清单中的值与页面展示完全一致，则 `mapping` 可为空
- 若页面下拉值与任务文件值不一致，则必须通过映射转换后再选择

#### `runtime`

用于描述运行时策略。

| 配置项 | 是否必填 | 含义 | v0.1 用途 |
|------|----------|------|-----------|
| `screenshot_on_error` | 是 | 失败时是否自动截图 | 建议设为 `true` |
| `log_level` | 是 | 日志级别 | 控制终端与日志输出粒度，v0.1 建议为 `INFO` |

### 7.1.3 配置设计约束

- v0.1 阶段不要继续扩充新的配置分组
- 页面选择器统一放在 `selectors` 中集中管理，不散落在代码里
- 凭据类信息不应硬编码在代码中，优先通过环境变量注入
- 若某个页面元素暂时无法稳定定位，可先在配置中留占位键，后续再补具体选择器
- `config.yaml` 中的字段命名应与代码中的配置模型保持一致，避免二次转换

### 7.1.4 当前已识别的登录页与首页入口选择器

基于当前已访问的登录页 URL、业务首页 URL 和前端 bundle 文案分析，v0.1 阶段建议优先使用 Playwright 风格选择器，而不是依赖过于脆弱的长 CSS 路径。

当前建议值如下：

| 配置项 | 建议值 | 说明 |
|------|--------|------|
| `username_input` | `input[placeholder="用户名"]` | 已从前端文案中确认存在“用户名”字段 |
| `password_input` | `input[placeholder="密码"]` | 已从前端文案中确认存在“密码”字段 |
| `login_button` | `button:has-text("登录")` | 已从前端文案中确认存在“登录”按钮 |
| `privacy_agreement_checkbox` | `text=同意并接受` | 登录页存在“同意并接受 用户隐私协议”相关控件时使用 |
| `upload_entry` | `text=上传档案附件` | 业务首页进入“上传档案附件”页面的入口，首版优先按可见文本定位 |

Playwright 实现时，推荐优先使用更稳的语义定位方式：

- `page.getByPlaceholder('用户名')`
- `page.getByPlaceholder('密码')`
- `page.getByRole('button', { name: '登录' })`
- `page.getByRole('button', { name: '上传档案附件' })`
- `page.getByText('上传档案附件')`

对于“用户隐私协议”复选框：

- 如果该复选框未勾选会阻止登录，则应纳入 `v0.1` 配置并在登录流程中处理
- 如果页面默认已勾选，或不影响点击登录，则可保留 selector 但在实现中按需使用
- 当前建议先在配置中保留 `privacy_agreement_checkbox`，避免后续登录流程因协议勾选状态阻断

对于“上传档案附件”入口：

- 当前环境只能访问 SPA 入口页面，无法直接拿到登录后渲染完成的最终 DOM
- 首版建议优先使用文本或按钮语义定位，不先依赖长 CSS 路径
- 若实际运行时发现页面中存在多个同名元素，应再结合容器区域或菜单层级补充更精确选择器

补充说明：

- 由于当前环境拿到的是前端 bundle 和页面入口 HTML，而非浏览器渲染后的最终 DOM，以上 selector 属于 v0.1 的高概率候选值
- 在实际编写 Playwright 脚本时，如页面渲染后的真实 DOM 与预期不一致，应以运行时验证结果为准进行微调

| 配置项 | 建议值 | 说明 |
|------|--------|------|

当前模板文件还包含第 2 行“字段说明行”，例如当前项目中的 [task.csv](/d:/3.其他工作/disney_file_upload/task.csv)：

任务唯一编号,人员编号/工号,人员姓名，可选,所属业务,业务类型,本地文件路径
TASK_001,EMP001,示例姓名,证明,收入证明,.\file\sample_1.pdf

程序处理规则建议如下：

- 第 1 行作为正式表头
- 若第 2 行内容明显为中文字段说明而非真实任务数据，则自动跳过

- Excel 模板也建议保持相同列顺序

需求要求：

- 导入前先校验必填字段

- 支持只校验不执行
- 支持指定部分任务执行

v0.1 版本说明：

- 关于“校验本地文件”，v0.1 版本仅校验任务清单中的必填项是否已填写
- 对于 `file_path` 字段，首版仅要求该字段非空，不校验文件是否真实存在、格式是否正确、大小是否超限、文件是否可读

## 7.7 提交确认与结果校验

- 当前记录的工号是否与任务清单一致
- 当前记录的所属业务是否与任务清单一致
- 当前记录的业务类型是否与任务清单一致
- 文件名是否与本地文件名一致或符合规则

v0.1 版本建议补充一条更贴近当前业务流程的成功校验规则：

- 系统提交成功后会自动跳转回档案管理列表页
- 列表按操作时间倒序排列，第一条记录应为最新上传的一条
- 程序需读取列表页第一条记录，并校验其是否为刚刚导入的档案记录

v0.1 推荐校验顺序：

1. 提交后确认页面已跳转回档案管理列表页
2. 读取列表第一页第一条记录
3. 使用当前任务信息比对第一条记录的关键字段
4. 若关键字段一致，则判定上传成功
5. 若关键字段不一致或无法读取，则判定失败或待人工复核

状态判定规则：

- 全部关键校验通过：`success`
- 任一关键校验失败：`failed`
- 无法自动判断：`need_review`

### 7.10.0 CLI 参数说明

为避免 POC 首版范围失控，CLI 参数分为“v0.1 POC 必须支持”和“后续版本再迭代”两层。

#### v0.1 POC 必须支持的最小参数集

首版建议只保留最关键、最能支撑端到端验证的参数：

- `validate`
- `run`
- `report`
- `--config`
- `--tasks`
- `--headed`

首版设计原则：

- 优先保证可以稳定跑通一批任务
- 优先保证操作人员容易理解和使用
- 不在首版引入过多筛选、静默、调试、批次控制参数
- 复杂参数能力放到后续版本，避免 POC 过度设计

首版推荐命令收敛为：

```bash
python main.py validate --config ./config.yaml --tasks ./task.csv
python main.py run --config ./config.yaml --tasks ./task.csv --headed
python main.py report --batch-id 20260402_001
```

说明：

- `validate`：首版保留，用于执行前校验任务清单和文件路径
- `validate`：首版保留，用于执行前校验任务清单中的必填项是否完整填写
- `run`：首版核心命令，用于真正执行浏览器自动化上传
- `report`：首版保留，用于查看批次执行结果
- `--config`：首版必须，决定系统地址、登录信息、选择器、日志路径等
- `--tasks`：首版必须，决定本次要执行的任务清单
- `--headed`：首版建议必须支持，便于观察浏览器行为和排查页面问题

#### 后续版本再迭代的参数

以下参数不建议纳入 v0.1 POC，可以在 v0.2 或之后逐步增加：

- `--headless`
- `--retry-failed`
- `--quiet`
- `--verbose`
- `--task-id`
- `--from-row`
- `--to-row`
- `--dry-run`

这些能力的共同特点是：

- 更偏“效率增强”或“运维增强”
- 不是打通首版业务闭环的必要条件
- 可以在 CLI 主流程稳定后再补，风险更低

#### 命令：`validate`

用于执行前校验，不真正启动完整上传流程。

示例：

```bash
python main.py validate --config ./config.yaml --tasks ./task.csv
```

参数说明：

- `validate`
  - 意图：执行预检查模式
  - 作用：校验配置文件、任务清单和必填字段完整性
  - 特点：不执行登录、不打开上传页面、不提交文件
  - v0.1 说明：对于本地文件，首版仅检查 `file_path` 是否已填写，不检查文件是否真实存在

- `--config`
  - 意图：指定配置文件路径
  - 作用：告诉程序本次运行使用哪个 `YAML` 配置文件
  - 示例：`--config ./config.yaml`

- `--tasks`
  - 意图：指定任务清单路径
  - 作用：告诉程序读取哪一个 `CSV` 或 `Excel` 文件作为上传任务输入
  - 示例：`--tasks ./task.csv`

#### 命令：`run`

用于真正执行浏览器自动化上传流程。

v0.1 POC 推荐示例：

```bash
python main.py run --config ./config.yaml --tasks ./task.csv --headed
```

参数说明：

- `run`
  - 意图：执行正式批量上传流程
  - 作用：启动浏览器自动化，按任务清单逐条执行登录、查询、上传、提交和校验

- `--config`
  - 意图：指定配置文件路径
  - 作用：加载系统地址、账号信息、选择器、映射、超时、日志路径等运行参数

- `--tasks`
  - 意图：指定任务清单路径
  - 作用：加载本次需要执行的上传任务列表

- `--headed`
  - 意图：以“有界面模式”运行浏览器
  - 作用：启动可见的浏览器窗口，让操作人员可以实时看到自动化执行过程
  - 适用场景：POC 验证、页面调试、人工介入、培训演示
  - 补充说明：开启后浏览器窗口会显示在桌面上

- `--headless`
  - 意图：以“无界面模式”运行浏览器
  - 作用：浏览器在后台执行，不显示窗口
  - 适用场景：稳定批处理、无人值守执行、后续服务化部署
  - 版本建议：放入后续版本，不纳入 v0.1 POC
  - 补充说明：后续实现时，`--headed` 与 `--headless` 应互斥

- `--retry-failed`
  - 意图：只重跑历史失败任务
  - 作用：根据数据库或上次执行记录，跳过已成功任务，仅重试失败项
  - 适用场景：批次中断后续跑、问题修复后的补跑
  - 版本建议：放入后续版本，不纳入 v0.1 POC

- `--quiet`
  - 意图：减少终端输出
  - 作用：仅输出关键错误和最终汇总，不展示每个步骤的详细过程
  - 适用场景：终端输出需要尽量简洁时
  - 版本建议：放入后续版本，不纳入 v0.1 POC

- `--verbose`
  - 意图：输出更详细的执行信息
  - 作用：展示更细粒度的步骤日志、等待信息、重试信息和状态变化
  - 适用场景：排查自动化问题、调试页面流程
  - 版本建议：放入后续版本，不纳入 v0.1 POC

- `--batch-id`
  - 意图：指定本次执行的批次号
  - 作用：用于结果追踪、日志归档、数据库关联
  - 适用场景：希望外部手工指定批次编号时
  - 首版建议：`run` 时可自动生成，`report` 时作为唯一查询参数保留
  - 补充说明：若不传，程序可自动生成，如 `20260402_001`

- `--task-id`
  - 意图：仅执行指定任务
  - 作用：从任务清单中筛选单条或少量任务执行
  - 适用场景：开发调试、问题复现、单任务补跑
  - 版本建议：放入后续版本，不纳入 v0.1 POC

- `--from-row`
  - 意图：从任务清单指定行开始执行
  - 作用：支持按行号截取后半段任务运行
  - 适用场景：批量任务很大时分段执行
  - 版本建议：放入后续版本，不纳入 v0.1 POC

- `--to-row`
  - 意图：执行到任务清单指定行为止
  - 作用：与 `--from-row` 搭配，执行一个行号区间
  - 版本建议：放入后续版本，不纳入 v0.1 POC

- `--dry-run`
  - 意图：模拟执行但不真正提交
  - 作用：可执行部分浏览器流程验证，但在最终提交前停止
  - 适用场景：上线前演练、页面流程核对
  - 版本建议：放入后续版本，不纳入 v0.1 POC

#### 命令：`report`

用于查看指定批次的执行结果汇总。

示例：

```bash
python main.py report --batch-id 20260402_001
```

参数说明：

- `report`
  - 意图：查看历史执行结果
  - 作用：从数据库或报告文件中读取指定批次的汇总信息

- `--batch-id`
  - 意图：指定要查看的批次号
  - 作用：查询对应批次的成功、失败、待复核、截图和错误摘要

#### 参数设计约束

- v0.1 POC 中，`run` 命令建议只要求 `--config`、`--tasks`、`--headed`
- 后续实现 `--headless` 后，需保证 `--headed` 与 `--headless` 互斥
- 后续实现 `--quiet` 与 `--verbose` 后，需保证两者互斥
- 后续实现 `--retry-failed` 后，`validate` 命令下不应允许使用该参数
- `report` 命令通常不需要 `--config` 和 `--tasks`
- 所有路径参数均应支持相对路径和绝对路径

### 7.10.1 终端实时执行展示

在执行 `run` 命令时，系统需在终端实时输出浏览器自动化执行过程，避免 CLI 工具表现为“无响应”或“黑盒运行”。

终端展示目标：

- 让操作人员知道当前正在执行哪个任务
- 让操作人员知道当前处于哪个自动化步骤
- 让操作人员看到成功、失败、重试、人工介入等状态变化
- 在批量执行时实时看到整体进度

建议展示内容至少包括：

- 当前批次号
- 当前任务 `task_id`
- 当前人员 `employee_id` / `employee_name`
- 当前步骤名称
- 步骤开始、完成、失败、重试状态
- 失败原因摘要
- 当前进度，例如 `3/20`
- 成功数、失败数、待复核数

建议输出风格：

- 以单行状态日志持续输出，不要求复杂 TUI
- 关键信息可使用固定前缀，如 `INFO`、`STEP`、`OK`、`WARN`、`ERROR`
- 每个任务开始和结束时输出清晰分隔信息
- 程序结束时输出批次汇总

示例：

```text
[INFO] batch=20260402_001 tasks=20 mode=headed
[STEP] task=TASK_001 employee=EMP001 step=open_login_page
[OK]   task=20260402_001 step=submit_login
[STEP] task=20260402_001 step=open_upload_page
[STEP] task=TASK_001 step=query_person keyword=EMP001
[OK]   task=20260402_001 step=confirm_person
[STEP] task=20260402_001 step=upload_file file="D:\3.其他工作\disney_file_upload\证明.pdf"
[OK]   task=20260402_001 step=submit_form
[OK]   task=20260402_001 step=verify_result status=success progress=1/20
[WARN] task=20260402_002 step=query_person retry=1 reason="result not unique"
[ERROR] task=20260402_003 step=upload_file reason="file not found"
[SUMMARY] total=20 success=18 failed=1 need_review=1 skipped=0
```

扩展要求：

- 终端输出应与日志系统复用同一套步骤事件模型，避免维护两套状态逻辑
- 支持 `--quiet` 模式，仅输出错误和最终汇总
- 支持 `--verbose` 模式，输出更细的步骤和页面等待信息

## 8. 业务流程

### 8.1 主流程

1. 用户准备配置文件、任务清单、待上传文件
2. 用户执行 CLI 命令
3. 系统校验输入是否合法
4. 系统初始化日志和数据库
5. 系统启动浏览器并登录外部平台
6. 系统逐条读取任务并执行上传流程
7. 系统输出成功、失败、待复核结果
8. 系统生成批次报告

### 8.2 单任务处理流程

1. 读取任务数据
2. 校验任务必填项是否完整填写
3. 查询目标记录
4. 进入记录详情页
5. 打开上传入口
6. 上传文件
7. 填写登记字段
8. 提交
9. 等待反馈并校验
10. 落库并写日志

### 8.3 异常处理流程

v0.1 统一异常处理口径如下：

1. 某一步骤失败
2. 记录失败步骤与错误原因
3. 自动截图
4. 写入 SQLite 和操作日志
5. 将当前任务标记为失败
6. 继续执行下一条任务

补充说明：

- v0.1 暂不实现“人工确认后继续”
- v0.1 暂不实现步骤级或任务级自动重试
- 若失败原因属于“无法唯一识别页面元素”或“结果无法自动判定”，首版同样按失败留痕处理，并继续后续任务

## 8.4 浏览器自动化详细执行流程

以下流程基于当前已提供的《DSN档案文件上传步骤.pdf》整理，作为 CLI POC 阶段的默认自动化路径。

### 8.4.1 登录流程

#### 页面入口

- 登录地址：`https://qa.magicjourney.shanghaidisneyresort.com/login?redirect_uri=%2F`

#### 执行步骤

1. 启动浏览器并打开登录页
2. 等待登录页加载完成
3. 定位用户名输入框并填写用户名
4. 定位密码输入框并填写密码
5. 点击`登录`按钮
6. 等待系统跳转到操作主界面

#### 成功判定

满足以下任一条件，可判定为登录成功：

- URL 从登录页跳转到业务首页
- 页面出现操作主界面的关键菜单
- 页面出现`上传档案附件`入口

#### 失败判定

出现以下情况之一，判定为登录失败：

- 页面提示用户名或密码错误
- 点击登录后长时间停留在登录页
- 页面出现异常提示、空白页或 4xx/5xx 错误页

#### 自动化注意点

- 登录后可能存在页面渲染延迟，需加入显式等待
- 密码、密钥类配置不得明文写入日志
- 如果后续出现验证码或二次验证，CLI POC 需切换为人工介入模式

### 8.4.2 进入档案上传页面

#### 执行步骤

1. 在操作主界面定位`上传档案附件`按钮或菜单
2. 点击进入档案上传操作界面
3. 等待上传页面加载完成

#### 成功判定

页面出现以下关键元素中的若干项：

- `上传方式`
- `选择人员`
- `所属业务`
- `业务类型`
- `上传文件`
- `提交`

#### 失败判定

- 点击后无跳转、无弹窗、无页面变化
- 页面缺少上传表单关键字段
- 页面加载超时

### 8.4.3 上传表单填写流程

#### 字段与动作

根据当前 PDF 和当前 `task.csv` 模板，上传页首版至少包含以下操作：

| 步骤 | 页面元素/动作 | 说明 |
|------|----------------|------|
| 1 | 上传方式 | 默认为`单个上传`，首版不调整 |
| 2 | 选择人员 | 点击右侧按钮进入人员选择弹窗 |
| 3 | 人员查询 | 优先使用 `employee_id` 查询，必要时辅以 `employee_name` |
| 4 | 查询按钮 | 点击查询/放大镜按钮 |
| 5 | 人员选中 | 在结果列表中勾选或选中对应人员 |
| 6 | 确认按钮 | 点击右下角`确认`返回上传页 |
| 7 | 所属业务 | 使用任务字段 `business_line` 选择对应业务 |
| 8 | 业务类型 | 使用任务字段 `business_type` 选择对应业务类型 |
| 9 | 上传文件 | 使用任务字段 `file_path` 选择本地文件 |

### 8.4.4 人员选择弹窗流程

#### 执行步骤

1. 点击`选择人员`右侧按钮
2. 等待人员选择弹窗出现
3. 在查询输入框中优先填入任务字段 `employee_id`
4. 若仅按 `employee_id` 未命中，且提供了 `employee_name`，则可切换为姓名辅助查询
5. 点击`查询`按钮
6. 等待结果列表刷新
7. 在结果列表中选中目标人员记录
8. 点击右下角`确认`
9. 返回上传页，并确认人员信息已成功回填

#### 成功判定

- 弹窗正常打开
- 查询结果列表出现候选记录
- 选中并确认后，上传页显示已选择的人员信息

#### 失败判定

- 弹窗未打开
- 查询后无结果
- 查询结果多条且无法自动唯一识别
- 点击确认后人员信息未回填

#### 自动化难点

- 弹窗可能为模态框，也可能嵌套 iframe
- 结果列表可能需要处理分页、勾选框或双击行
- 当查询结果多条时，需定义更明确的自动匹配规则

#### 建议匹配策略

首版按以下优先级匹配人员：

1. 精确匹配 `employee_id`
2. 精确匹配 `employee_name` + `employee_id`
3. 若仍有多条，则标记为`need_review`

### 8.4.5 所属业务与业务类型选择

#### 执行步骤

1. 在上传页定位`所属业务`字段
2. 使用任务字段 `business_line` 或映射值进行选择
3. 定位`业务类型`字段
4. 使用任务字段 `business_type` 或映射值进行选择

#### 成功判定

- 页面中已显示所选的业务值
- 字段值可在提交前再次读取确认

#### 风险点

- 下拉框可能是原生 select，也可能是自定义组件
- `业务类型`可能受`所属业务`联动影响，需要等待可选项刷新

### 8.4.6 文件上传流程

#### 执行步骤

1. 点击`上传文件`
2. 触发文件选择控件
3. 注入任务字段 `file_path` 对应的本地文件路径
4. 等待上传完成提示、文件名回显或附件列表更新

#### 成功判定

满足以下任一条件，可视为上传动作成功：

- 页面显示已选择或已上传的文件名
- 页面出现上传完成提示
- 附件区域出现新增文件记录

#### 失败判定

- 文件控件无法触发
- `file_path` 未填写
- 页面提示格式不支持或上传失败
- 长时间无上传完成反馈

### 8.4.7 提交与确认流程

#### 执行步骤

1. 在上传页点击右上角`提交`
2. 等待系统弹出确认框
3. 在确认框中点击`确定`
4. 等待系统提交完成并跳转回`档案管理`列表页

#### 成功判定

- 成功返回`档案管理`列表页
- 列表第一页第一条记录为刚完成上传的材料记录

#### 失败判定

- 点击提交后无响应
- 未出现确认框
- 点击确定后系统报错
- 未回到列表页

### 8.4.8 上传结果校验流程

根据当前 PDF 和当前业务规则，v0.1 首版采用“返回列表页后校验第一条记录”的方式做上传成功验证。

#### 建议校验项

- 当前页面已回到`档案管理`列表页
- 列表按操作时间倒序显示
- 第一条记录即为最新一条上传记录
- 第一条记录的人员信息与当前任务一致
- 第一条记录的所属业务与任务一致
- 第一条记录的业务类型与任务一致
- 第一条记录的附件名称与本地文件名称一致或满足命名规则

#### v0.1 推荐最小校验口径

考虑到首版以 POC 为主，v0.1 明确采用以下两项作为最小成功校验口径：

- 第一条记录的 `employee_id`
- 第一条记录的 `business_type`

若以上两个字段与当前任务一致，则认为“刚才操作导入的档案记录”已成功出现在列表首条，可判定本次上传成功。

#### v0.1 具体比对规则

程序在跳转回列表页后，应执行以下判断：

1. 读取列表第一页第一条记录
2. 提取该记录中的 `employee_id`
3. 提取该记录中的 `business_type`
4. 与当前任务中的 `employee_id`、`business_type` 做精确匹配

判定逻辑：

- 若第一条记录的 `employee_id` 等于当前任务的 `employee_id`
- 且第一条记录的 `business_type` 等于当前任务的 `business_type`
- 则判定该记录为刚完成上传的记录，本次上传结果记为 `success`

否则：

- 若任一字段不一致，则判定为 `failed`
- 若列表字段无法读取、页面结构无法识别或列表未正常加载，则判定为 `need_review`

#### 说明

- `employee_name` 不作为 v0.1 成功校验主键，仅用于人员查询时的辅助匹配
- `business_line`、附件名称等字段在首版中可作为辅助日志信息记录，但不作为最终成功判定的必要条件
- 后续版本可将附件名称、所属业务等字段加入多字段联合校验

#### 判定规则

- 关键字段全部一致：`success`
- 关键字段部分不一致：`failed`
- 页面结构无法读取或记录无法唯一确认：`need_review`

### 8.4.9 建议的步骤级状态机

为便于日志记录和断点续传，建议内部将每条任务拆成以下步骤状态：

1. `open_login_page`
2. `submit_login`
3. `open_upload_page`
4. `open_person_selector`
5. `query_person`
6. `confirm_person`

## 8.5 自动化异常点与后续扩展设计

结合当前 PDF 和 CLI POC 目标，以下场景在后续版本中可考虑扩展更复杂的处理策略；但在 v0.1 中统一按“失败留痕并继续下一条”处理。

### 8.5.1 后续版本可扩展处理的场景

- 登录后出现验证码、短信码或二次确认
- 人员查询结果多条，无法自动唯一匹配
- 所属业务/业务类型配置值在页面中不存在
- 页面出现未知弹窗或权限提示
- 上传成功但返回列表页后无法自动确认第一条记录

### 8.5.2 CLI POC 处理策略

- 默认自动执行
- 命中上述场景时记录失败原因
- 自动截图并写入日志、数据库
- 将当前任务标记为失败后继续下一条
- v0.1 不实现 `manual_intervention`

### 8.5.3 不建议首版自动处理的场景

- 绕过验证码
- 复杂图片识别
- 非标准安全控件逆向
- 未授权的接口调用

## 10. 非功能需求

### 10.1 稳定性

- 页面元素查找需带超时与重试机制
- 会话失效需可自动重新登录
- 长时间批处理不能因单点失败中断

### 10.2 可维护性

- 页面选择器配置化
- 字段映射配置化
- 日志清晰，模块职责明确
- CLI 层与核心流程层解耦

### 10.3 可观测性

- 全链路日志
- 关键步骤截图
- 批次级汇总报表
- 终端实时步骤输出与进度展示

### 10.4 安全性

- 凭据不写死在代码中
- 支持环境变量注入敏感信息
- 日志中避免明文输出密码和密钥

### 10.5 合规性

- 仅在授权范围内使用
- 自动化动作不绕过系统控制
- 保留人工接管与停止能力

## 13. 风险与应对

| 风险 | 描述 | 应对 |
|------|------|------|
| 页面结构变化 | DOM、按钮文案、弹窗逻辑变化导致自动化失效 | 选择器配置化，关键节点增加兜底策略 |
| 登录流程变化 | 出现验证码、二次校验、短信验证等 | 首版支持人工介入，不承诺自动绕过 |
| 任务数据质量差 | 字段缺失、映射错误、任务清单填写错误 | 增加 `validate` 模式提前拦截 |
| 上传控件特殊 | 非标准 input 上传控件难自动化 | 在 POC 阶段先验证目标页面可操作性 |
| 网络或会话异常 | 长批次过程中掉线、超时 | 自动重试、自动重登、失败留痕 |
